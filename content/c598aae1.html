一、概念

RunLoop 运行循环，是与线程相关的基础架构的一部分，它是一个对象，用来管理需要处理的事件和消息，并提供入口函数来执行事件循环时的逻辑。运行循环的目的是在有工作要做时让线程保持忙碌，没有工作时让线程休眠。

每个线程，包括应用程序的主线程，都有一个相关的运行循环对象。线程与 RunLoop 之间存在一对一的关系，该关系存储在一个全局的 Dictionary 中。主线程的运行循环作为应用程序启动过程的一部分，自动设置并在主线程上运行。非主线程的运行循环需要显示运行，在创建线程时，并没有 Runloop ，如果不主动获取，它永远不会存在。Runloop 在第一次获取时创建，除了主线程之外，只能在线程内部获取 RunLoop。

二、RunLoop 存在的意义

通常一个线程一次只能执行一个任务，任务执行完毕时，退出线程。RunLoop 是一个运行循环，内部是一个 do...while 循环，它在线程中执行可以保持线程持续运行不退出，以”接收消息->等待->处理”的循环模式持续运行，直到循环结束。实现此模式的关键是管理事件/消息时，如果在不处理消息时，使线程休眠以免资源占用，在消息到达时唤醒线程。主要来说作用有以下几点：

  • 保持程序运行
  • 接收用户输入
  • 监听事件
  • 节省CPU时间
  • 调用解耦

三、事件源

在OS X和iOS系统中，提供了两种 RunLoop 对象 ：

CoreFoundation 框架内的 CFRunLoopRef ，该框架为纯C函数提供的API，所有API都是线程安全的。
Foundation 框架内的 NSRunLoop ，NSRunLoop 是基于 CFRunLoopRef 的封装，提供了面相对象的API，是非线程安全的。

NSRunLoop 可以通过 [[NSRunLoop currentRunLoop] getCFRunLoop] 转换为 CFRunLoopRef 。

RunLoop 从两种不同类型的事件源接收消息：输入源( Input Sources )：传递异步事件，通常是来自另一个线程或其他应用程序的消息。计时器源(Timer Sources)：传递同步事件，这些事件是在计划的时间或重复的时间间隔发生。此外，运行循环还会生成有关运行循环行为的通知，注册的运行循环观察者可以接收这些通知，并使用它们在线程中进行其他处理。

1.输入源(Input Sources)

输入源包括三种类型的源：

  • 基于端口的输入源：Cocoa 和 Core Foundation 提供了用于线程或进程间通信的 Mach Port ，只需创建一个端口对象，然后将该对象添加到运行循环即可。在 Core Foundation 中需要手动创建端口及运行循环。

  • 选择器源：选择器源在执行后会将其自身从运行循环中删除。执行选择器源时，目标线程必须有活动的运行循环。每次循环时，运行循环都会将所有排队的选择器任务进行执行调用，而不是每次执行一个。选择器有以下几种：

2.计时器源

计时器源在将来的预设时间将事件同步传递到线程，是线程通知自己执行某事的一种方式。在 Cocoa 中，使用 NSTimer 创建计时器来加入 RunLoop ，在 Core Foundation 中使用 CFRunLoopTimerRef 类型。NSTimer 是 CFRunLoopTimerRef 的简单扩展。

虽然计时器基于时间创建，但是它并非实时机制，计时器与运行循环的特定模式相关联。如果计时器不在运行循环当前正在监视的模式下，则只有当以计时器支持的一种的模式运行循环后，才会触发。如果计时器添加到运行循环时，运行循环正在执行处理程序，则计时器将要等到下一次运行循环调用。如果没有运行循环，则计时器不会触发。

对于重复执行的计时器，会根据计划触发时间(而不是实际触发时间)自动重新计划自身。比如：如果计时器计划在特定时间触发，然后每隔5秒触发一次，则即使实际触发时间被延迟，计划触发时间也将始终落在原始的5秒时间间隔上。如果触发时间延迟太多，以致错过了一个或多个计划触发时间，则计时器将在错过的时间段内只执行一次。

3.观察者

观察者在运行循环本身执行期间的特定位置触发。需要使用 Core Foundation 框架中的 CFRunloopObserverRef 创建，并通过 CFRunLoopAddObserver 将观察者添加到运行循环中。运行循环可以指定使用一次或多次使用。一次触发的观察者在触发后将自己从运行循环中删除，重复执行的观察者将处于连接状态。每个观察者都包含一个回调(函数指针)，当运行循环的状态改变时，观察者可以通过回调来接收改变。

四、循环模式

运行循环模式是要监视的输入源和计时器以及要通知的运行循环观察者的集合。每次运行循环，都可以显示或隐式指定特定的模式。在运行循环的整个过程中，仅监视与该模式相关的源，并允许其传递时间。

一个 RunLoop 包含多个模式，每个模式都包含多个 Source/Timer/Observer 模式项 (上面介绍的三种)。每次调用 RunLoop 只能指定一种模式，如果需要切换模式，则只能退出循环，然后重新指定需要的模式。一个模式项可以同时添加到多个模式，但重复添加到同一模式将不起作用。没有模式项(事件源)的 RunLoop 将不能进入循环。

模式根据事件的来源进行区分，而不是事件的类型。输入源将事件异步传递到线程，事件的来源取决于输入源的类型。基于端口的输入源监视应用程序的 Mach 端口，自定义输入源监视定制源。两种源唯一的区别是信号发送的方式。基于端口的源由系统内核自动发出信号，自定义源需要手动发出信号。

系统默认已注册5种模式：

kCFRunLoopDefaultMode 是 RunLoop 启动时的默认模式。UITrackingRunLoopMode 跟踪 ScrollView 状态。当创建一个 NSTimer 并将其添加到默认模式时，NSTimer 将获得回调。但当滑动 ScrollView 相关控件时，RunLoop 会将模式切换为 UITrackingRunLoopMode ，此时 NSTimer 将不会再被调用，也不影响滑动操作。如果想在两种模式下都被回调，可以将 NSTimer 同时加入到两种模式之中，也可以将 NSTimer 添加到 kCFRunLoopCommonModes ，CommonModes 并不是一种实模式，而是 kCFRunLoopDefaultMode 与 UITrackingRunLoopMode，此两种模式均已标记为通用( Common )属性。当 RunLoop 内容发生更改时，RunLoop 会自动将 CommonModes 中的 Source/Timer/Observer 与带有 Common 标志的所有模式同步。

添加观察者

CFRunLoopObserverContext 用来创建观察者的上下文环境，它是一个结构体，包含程序定义的数据和回调结构。version 为结构体版本号，必须为0；info 为任意指针，可通过此指针传递上下文中定义的所有回调；retain 保留任意指针，可以为NULL；release 释放任意指针，可以为NULL；copyDescription 任意指针的副本描述回调，也是一个函数指针，可以为NULL。

CFRunLoopObserverCreate 创建观察者，有6个参数，allocator 为新对象的内存分配器，传递 NULL 或 kCFAllocatorDefault ; activities 为观察者要观察的运行循环标志，上面有介绍，可以看CFRunLoopActivity ，如需观察多个状态，可使用按位或进行组合；repeats 标志只调用一次还是每次都调用观察者；order 优先级，苹果建议使用0；callout 回调函数；context 上下文信息，可以为NULL。

CFRunLoopAddObserver 添加观察者到指定 RunLoop ，三个参数，rl 为要添加到的运行循环；observer 要添加的观察者；mode 循环模式。
